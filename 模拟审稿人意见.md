# 模拟审稿人意见

> 论文题目：面向服务机器人的异构常识知识库构建与检索增强指令语义理解  
> 审稿类型：模拟外审意见 / 投稿前内部评审  
> 建议结论：**大修后再审（Major Revision）**  
> 适用用途：用于论文修改、实验补充规划、答辩前自查和投稿前质量控制

---

## 一、总体评价

本文面向中文服务机器人交互场景中的反讽、隐喻和网络流行语理解问题，提出构建三源异构常识知识库，并通过检索增强方式将外部知识注入 BERT-Base-Chinese 模型，以提升中性、正面、反讽和愤怒四分类任务的性能。论文选题具有明确的应用背景，问题定位较具体，方法路线整体清晰，实验结果显示检索增强模型较纯文本 BERT 基线取得明显提升，说明外部知识对服务机器人非字面语义理解具有一定价值。

论文的主要优点包括：

1. 选题具有实际应用意义，能够对应服务机器人在人机交互中的真实问题；
2. 方法设计具有较强工程可实现性，采用知识库构建、FAISS 检索和 BERT 分类的轻量化路线；
3. 实验已报告多随机种子结果和混淆矩阵分析，初步体现了方法稳定性；
4. 三源知识库能够覆盖结构化常识、网络流行语和服务场景表达，具有一定创新性。

但是，当前稿件距离正式投稿仍存在若干关键不足，尤其是数据来源透明性、三源知识库必要性证明、检索策略消融、真实场景泛化能力和外部基线对比等方面仍需补强。若不补充这些实验，论文容易被审稿人质疑为“工程组合有效，但机制证据不足”。

## 二、主要问题

### 问题 1：数据来源、标注过程和数据可靠性说明不足

当前论文说明使用固定独立测试集，并报告了多种子实验结果。但数据集的来源、生成方式、标注规则和人工复核流程仍需进一步说明。

需要明确回答以下问题：

1. 训练集、验证集和测试集是否全部由 LLM 生成；
2. 若使用 LLM 生成数据，生成 Prompt、温度、过滤规则和重复检查方式是什么；
3. 四类标签“中性、正面、反讽、愤怒”的判定标准是什么；
4. 是否进行人工抽检；
5. 是否报告人工标注一致性；
6. 训练集、验证集、测试集之间是否存在文本重叠或近重复；
7. 测试集是否能够代表真实服务机器人交互表达。

当前最大风险在于：如果训练和测试数据主要来自 LLM 合成，审稿人可能质疑模型学到的是生成器风格，而不是真实用户交互中的反讽理解能力。

#### 建议修改

在“实验设计”中新增“数据集构建与质量控制”小节，至少包括：

- 数据来源说明；
- 四分类标签定义；
- train/dev/test 切分方式；
- 完全文本去重；
- 近重复检查；
- 人工抽检比例；
- 类别分布；
- 测试集不参与训练和阈值调参的说明。

#### 推荐新增表格

| 标签 | 含义 | 判定标准 | 示例 |
|---|---|---|---|
| 中性 | 客观描述或普通请求 | 无明显情绪反转 | “请帮我送到前台” |
| 正面 | 明确赞扬或满意 | 字面与真实意图均为正向 | “服务很及时” |
| 反讽 | 字面正向但真实负向 | 存在语义反转或语境冲突 | “你这速度真快啊” |
| 愤怒 | 直接负面抱怨 | 无需反讽即可识别强烈不满 | “太差了，我要投诉” |

### 问题 2：三源知识库的必要性尚未充分证明

论文提出融合 ConceptNet 中文子图、网络流行语和 LLM 服务场景表达，构建三源异构知识库。但当前实验主要验证“使用完整知识库是否优于不使用知识库”，尚不能证明三个知识源分别具有不可替代作用。

如果缺少逐源消融，审稿人可能认为“三源异构”只是工程堆叠，而非经过实验证实的有效设计。

#### 必须补充的实验：逐源消融实验

建议补充如下实验组：

| 实验组 | 知识库配置 | 实验目的 |
|---|---|---|
| Baseline | 不使用知识库 | 纯文本基线 |
| Full KB | ConceptNet + 网络流行语 + LLM 服务表达 | 主方法 |
| w/o ConceptNet | 去除 ConceptNet | 验证结构化常识贡献 |
| w/o Slang | 去除网络流行语 | 验证网络亚文化知识贡献 |
| w/o LLM-Service | 去除 LLM 服务表达 | 验证服务场景领域知识贡献 |
| Only ConceptNet | 仅使用 ConceptNet | 分析单源结构化常识效果 |
| Only Slang | 仅使用网络流行语 | 分析单源网络表达效果 |
| Only LLM-Service | 仅使用 LLM 服务表达 | 分析单源服务领域知识效果 |

#### 推荐报告指标

- Accuracy；
- Macro F1；
- Neutral Recall；
- Sarcasm Recall；
- Anger Recall；
- 多种子均值和标准差，若算力不足，可先在 seed=1 上完成消融。

#### 论文中可写结论

> 逐源消融结果显示，去除任一知识源均导致不同程度的性能下降，说明结构化常识、网络流行语和服务场景表达在服务机器人指令语义理解中具有互补作用。

### 问题 3：双路检索策略缺少分类级消融

论文报告了检索器层面的 Precision、Recall 和 F1，显示双路检索优于单一路径。但检索器层面的效果并不能直接证明最终分类效果也最好。

当前还需要补充“分类级检索策略消融”，证明双路检索确实提升最终任务表现。

#### 必须补充的实验：检索策略消融

建议补充如下实验组：

| 实验组 | 检索方式 | 目的 |
|---|---|---|
| Baseline | 不检索 | 纯文本基线 |
| Random-Knowledge | 随机拼接知识 | 排除输入变长伪增益 |
| Exact-only | 仅精确子串匹配 | 验证高精度路径效果 |
| Vector-only | 仅 FAISS 向量检索 | 验证语义召回路径效果 |
| Dual-path | 精确匹配 + FAISS | 主方法 |
| No-threshold FAISS | 向量检索不设阈值 | 验证阈值过滤作用 |
| τ sweep | 不同阈值 | 验证 τ=0.72 的合理性 |

#### 推荐阈值设置

建议至少测试：

```text
τ = 0.50, 0.60, 0.70, 0.72, 0.80, 0.90
```

#### 推荐报告指标

| 方法 | Acc | Macro F1 | Sarcasm Recall | Neutral Recall |
|---|---:|---:|---:|---:|
| Baseline |  |  |  |  |
| Random-Knowledge |  |  |  |  |
| Exact-only |  |  |  |  |
| Vector-only |  |  |  |  |
| Dual-path |  |  |  |  |

#### 论文中可写结论

> Random-Knowledge 未带来同等提升，说明性能提升并非由输入长度增加导致，而主要来自相关知识注入。Exact-only 精度较高但覆盖有限，Vector-only 召回更强但易引入噪声，Dual-path 在准确率和反讽召回率上取得更优平衡。

### 问题 4：外部模型基线不足

当前论文主要对比 BERT 与 BERT+RAG。该对比可以证明检索增强对 BERT 有效，但不足以说明方法具有较强竞争力，也无法判断方法是否依赖特定模型。

建议增加更强中文预训练模型或常见分类结构作为基线。

#### 建议补充的基线

| 模型 | 作用 |
|---|---|
| BERT-Base-Chinese | 当前主基线 |
| BERT + Random Knowledge | 排除输入变长伪增益 |
| BERT + RAG | 当前主方法 |
| RoBERTa-wwm-ext | 中文强基线 |
| RoBERTa-wwm-ext + RAG | 验证方法可迁移性 |
| MacBERT | 中文强基线 |
| MacBERT + RAG | 验证方法可迁移性 |
| BERT+BiLSTM | 传统增强结构对比 |
| BM25 检索 + BERT | 检索方法替代基线 |

#### 最小补充方案

如果时间有限，至少补充：

1. RoBERTa-wwm-ext；
2. MacBERT；
3. BERT + Random Knowledge；
4. BM25 或 Exact-only 检索增强。

#### 论文中可写结论

> 在不同中文预训练模型上，检索增强均取得稳定提升，说明本文方法并非依赖特定 BERT 实现，而是来自外部异构常识知识的有效注入。

### 问题 5：真实服务场景验证不足

论文题目和应用背景均指向“服务机器人”，但当前主要实验仍是离线文本分类。如果没有真实或半真实服务机器人交互数据，应用场景支撑会不足。

#### 建议补充真实服务场景小样本测试集

建议构建 100–300 条真实或半真实服务机器人交互样本，来源可以包括：

1. 实验室同学模拟真实用户表达；
2. 问卷采集；
3. 服务投诉评论改写为机器人交互语句；
4. 餐厅、酒店、医院、智能家居等场景脚本；
5. 人机交互实验中的语音转写文本。

#### 推荐标注字段

每条样本建议包含：

```json
{
  "text": "你这机器人可真聪明，撞墙了还往前冲",
  "scene": "室内导航",
  "label": 2,
  "literal_polarity": "positive",
  "real_intent": "complaint",
  "has_sarcasm": true
}
```

#### 标注要求

- 至少 2 名人工标注者；
- 最好 3 名人工标注者；
- 报告 Cohen’s Kappa 或 Fleiss’ Kappa；
- 报告类别分布；
- 只作为测试集，不参与训练。

#### 推荐报告表格

| 数据集 | 样本数 | Baseline Acc | +RAG Acc | Sarcasm Recall |
|---|---:|---:|---:|---:|
| 固定测试集 | 400 |  |  |  |
| 真实服务场景集 |  |  |  |  |

#### 论文中可写结论

> 在真实服务场景小样本测试集上，检索增强模型仍优于纯文本基线，说明异构知识库并非仅适配合成数据，而对实际服务交互表达具有一定泛化能力。

若真实集效果下降，也可写：

> 真实场景测试集上的提升幅度低于固定测试集，说明合成数据与真实用户表达之间仍存在分布差异，后续需要进一步扩充真实交互语料。

---

### 问题 6：愤怒类样本过少，结果解释需谨慎

当前混淆矩阵显示愤怒类样本数量较少，导致该类 Recall 和 F1 不稳定。论文不应对愤怒类做过强结论。

#### 建议修改

1. 报告每一类测试样本数；
2. 单独说明愤怒类为低频类别；
3. 主结论更多围绕中性和反讽类展开；
4. 后续扩充真实投诉和强负面表达样本；
5. 可尝试类别重采样、类别权重或 Focal Loss。

#### 论文中可写结论

> 由于愤怒类在测试集中样本数量较少，该类别指标具有较高方差。因此，本文主要关注中性与反讽类别的性能变化，并将低频强负面表达识别作为后续优化方向。
